# ADR 004: 为什么不自建全功能网关

**日期**: 2026-06-28  
**状态**: 已采纳

## 背景

TraceLoop 需要拦截 LLM 调用以采集 trace。一个自然的想法是「不如做一个完整的 LLM 网关」——统一路由、限流、计费、采集、评测。需要决定是否走这条路。

## 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 自建全功能网关 | 一步到位；功能完整 | 工程范围远超预期；LiteLLM 供应链事件后自建网关信任成本极高 |
| 轻量代理，对接现有网关 | 聚焦核心；工程可控；用户可搭配自己的网关 | 采集层功能受限；路由/限流依赖外部 |

## 决策

**不做全功能网关，只做轻量采集代理**

## 理由

1. **LiteLLM 供应链攻击事件**（2026年3月）：LiteLLM v1.82.7/v1.82.8 被植入凭据窃取代码，使得自建 LLM 网关的信任成本大幅提高。用户对「又一个 LLM 代理」天然不信任。
2. **工程范围失控**：网关需要处理路由匹配、多模型适配、流式安全护栏、Redis Lua 原子预算等，每一项都是独立工程。如果什么都做，12 周连 MVP 都出不来。
3. **正确的策略是聚焦**：只做采集和评测，路由和限流交给 LiteLLM/OneAPI。用户已经在用网关的，TraceLoop 无缝切入；没用网关的，推荐他们搭配使用。
4. **差异化清晰**：Langfuse 做观测、LiteLLM 做路由、promptfoo 做 prompt 测试、我们做 CI 门禁。不试图替代谁，做好自己那块。

## 权衡

- 采集层功能受限：不支持根据模型自动路由，不支持 token 预算是好事（不需要维护那个复杂度）。
- 用户如果没装网关，需要额外部署。但这正是「聚焦」的价值——我们不需要为所有人解决所有问题。
- 后续如果需要内置更复杂的路由逻辑，可以通过插件机制扩展，而不是改核心架构。
