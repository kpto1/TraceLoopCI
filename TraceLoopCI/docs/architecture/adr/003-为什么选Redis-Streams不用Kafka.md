# ADR 003: 为什么选 Redis Streams 不用 Kafka

**日期**: 2026-06-28  
**状态**: 已采纳

## 背景

Trace 写入需要异步化：API 收到请求后不能等数据库写完再返回。需要消息队列做「前台收单→后台写库」的解耦。

## 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| Redis Streams | 已经依赖 Redis；Python 库成熟；消费者组机制够用；轻量 | 消息持久化和回溯不如 Kafka |
| Kafka | 吞吐量最强；消息可回溯；企业级持久化 | 需要额外运维；对中小团队太重；Python 客户端体验一般 |
| RabbitMQ | 功能完善；社区大 | 比 Redis 重；运维成本不比 Kafka 低多少 |
| 直接写数据库 | 最简单 | API 响应延迟随写入量线性增长，不可接受 |

## 决策

**Redis Streams**

## 理由

1. 项目已经依赖 Redis（做缓存和速率限制），不需要额外引入中间件。docker-compose 多一行就能跑。
2. Redis Streams 的消费者组机制满足需求：多个 worker 自动负载均衡，消息确认（XACK）保证不丢数据，stream 长度限制（MAXLEN）防止内存暴涨。
3. Python 的 redis 库（redis-py）对 Streams 支持完善，不需要额外的客户端依赖。
4. 万级日调用量的吞吐量，Redis 轻松应对。不需要 Kafka 级别的百万 QPS 能力。

## 权衡

- 消息持久化不如 Kafka：Redis 挂了可能丢最近的消息（默认内存存储，可配置 AOF 持久化但不完美）。对于评测场景，丢几条 trace 不是致命问题。如果未来需要严格的消息不丢失，可以加一个降级路径：API 同步写一条 backup 到日志文件。
- 消费者重启后从上次位置继续消费，但如果 Redis 重启且没开启持久化，stream 数据会丢失。生产部署时建议开启 Redis AOF。
