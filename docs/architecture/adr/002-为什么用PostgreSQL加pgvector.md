# ADR 002: 为什么用 PostgreSQL + pgvector

**日期**: 2026-06-28  
**状态**: 已采纳

## 背景

TraceLoop 需要存储两类数据：(1) Trace 记录（结构化数据：模型名、token 数、延迟、成本等），(2) 向量数据（用于 RAG 场景的文档检索评测）。需要选一个数据库方案。

## 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| PostgreSQL + pgvector | 一个数据库搞定；运维成本低；SQL 生态成熟 | 向量检索性能不如专用数据库 |
| PostgreSQL + Milvus | 向量检索性能最强 | 两套数据库运维；MVP 阶段过于复杂 |
| MySQL | 最熟悉的结构化数据库 | 无向量检索能力 |

## 决策

**PostgreSQL 16 + pgvector 插件**

## 理由

1. 一个数据库搞定业务数据 + 向量检索。不需要维护两套存储，部署和备份都简单。
2. pgvector 对 MVP 阶段的向量检索量够用。评估场景的文档数量通常是百到千级，不是百万级。
3. SQLAlchemy 支持好。ORM 操作结构化数据，原生 SQL 做向量检索，两者可以在同一次查询里混用。
4. Docker 镜像 `pgvector/pgvector:pg16` 开箱即用，几行 compose 配置就起来了。

## 权衡

- 向量检索性能上限不如 Milvus/Weaviate。百万级以上的向量检索会明显变慢。考虑到 MVP 阶段评测用的文档规模（百到千级），pgvector 完全够用。
- 后续如果做知识库级别的评测（万级以上文档），可以考虑把向量部分迁移到专用向量数据库。数据模型设计时已预留接口抽象层。
